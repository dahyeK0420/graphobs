# 0.3.2

`0.3.2` deepens the internal architecture, giving several cross-cutting
concerns a single home, with no change to the package-root interface.

## Highlights

- Runtime read enforcement moves into `contracts/validation.py` beside write
  validation (`enforce_undeclared_reads`), and the `graphobs.langgraph.read_audit`
  module is removed; reads and writes now share one entry over
  `contracts.conformance`.
- Each `NodeContractMode` resolves to one complete execution plan (pass-through,
  read tracking, read action, write action) in a single table, replacing a
  lookup table plus a second derivation step that disagreed on whether strict
  execution audits reads.
- `ProjectionPolicy` owns its projection behavior directly. The `project_state`
  free function and the redundant `ProjectionPolicyLike` / `ContractProjection`
  structural protocols are removed in favor of the concrete `Contract` interface.
- The correlation-conflict rule is defined once as `reconcile_correlation` in
  `logging/context.py`, and `GraphLogCallback` owns lifecycle timing and emission
  directly (the `LifecycleLogEmitter` indirection is gone; payload assembly
  stays as pure helpers).
- The input-span projection is derived once inside `instrument_contract_run`,
  removing the duplicated `ContractRunSpec.span_input` field.

## Migration Notes

These changes are internal refactors and the package-root interface is
unchanged. Code that imports lower-level submodules should note:

- `graphobs.langgraph.read_audit` is removed. Import `enforce_undeclared_reads`
  from `graphobs.contracts.validation`.
- `graphobs.contracts.projection` no longer exports `project_state`,
  `ProjectionPolicyLike`, or `ContractProjection`. Use `ProjectionPolicy.project`
  and type against the concrete `graphobs.contracts.models.Contract`.
- `graphobs.langgraph.execution.ContractRunSpec` no longer has a `span_input`
  field; the public input span payload is derived from the contract.
